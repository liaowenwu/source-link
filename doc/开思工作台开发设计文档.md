## 2. 模块目标
“开思工作台”是面向报价处理人员的业务工作台，核心目标如下：

1.  提供 **接单服务监控能力**
2.  提供 **报价单全流程处理能力**
3.  提供 **自动补价 / 手动补价 / 自动提交 / 手动提交** 的灵活模式 
4.  提供 **异常提醒与动态时间线**
5.  提供 **用户维度的抓价配置能力**
6.  提供 **日志可追踪、状态可追溯、处理可回放** 的系统能力 

---

## 3. 核心架构设计
## 3.1 架构说明
本次调整后的核心架构采用：

+  前端负责操作入口与状态展示 
+  Flask 负责接单服务的启动/停止指令，以及队列后台线程管理 
+  Redis 负责生产者-消费者队列解耦 
+  Flask 通过 WebSocket/HTTP 将处理状态和日志同步到后端服务 
+  后端服务负责持久化报价单、明细、日志、提交记录，并向前端展示 

### 核心原则
1. **Flask 不承担复杂业务状态持久化**
2. **Flask 只负责启动/停止、轮询、入队、消费、同步状态**
3. **Redis 作为队列中枢**
4. **后端服务作为业务主数据中心**
5. **前端只调用后端服务和少量 Flask 控制接口**

## 4. 核心接单流程设计
## 4.1 流程概述
### 功能链路
1.  前端调用 Flask 服务执行 **开启/停止接单**
2.  Flask 主线程轮询查询开思待报价报价单 
3.  将新报价单放入 **队列1**
4.  多线程消费者处理队列1，完成抢单与详情采集，并同步状态到后端 
5.  根据配置判断是否进入 **队列2**
6.  队列2执行补价逻辑 
7.  根据配置判断是否进入 **队列3**
8.  队列3执行提交报价单逻辑 
9.  完成后清理 Redis 数据，并将最终结果写入后端数据库

## 4.3 队列解耦设计
### 队列1：抢单与采集详情
职责：

+  从开思平台抓取待报价单 
+  抢单 
+  采集报价单详情 
+  将报价单主表、明细、执行日志同步到后端 

分支逻辑：

+  若开启 **手动补价模式**
    -  不进入队列2 
    -  等待前端人工补价 
    -  Flask 需提供“手动补价接口” 
+  若未开启手动补价模式 
    -  进入队列2 

### 队列2：补价逻辑
职责：

+  读取报价单详情 
+  根据用户抓价配置，执行价格抓取和价格计算 
+  将补价结果同步到后端 
+  判断是否进入队列3 

分支逻辑：

+  若开启 **自动提交**
    -  自动进入队列3 
+  若未开启自动提交 
    -  不进入队列3 
    -  等待前端调用提交接口 

### 队列3：提交报价单
职责：

+  读取待提交报价单 
+  执行开思平台报价提交 
+  持久化提交结果 
+  清理 Redis 中间数据 
+  更新最终状态

+  最近抓取时间

+  启动接单 
+  关闭接单

+  状态卡片每 3~5 秒自动刷新 ，自动修改数字，页面上不要有刷新的感觉

## 6. 业务状态设计
---

### 6.1 接单服务状态
建议定义：

+ `INIT`：未启动 
+ `STARTING`：启动中 
+ `RUNNING`：运行中 
+ `STOPPING`：停止中 
+ `STOPPED`：已关闭 
+ `ERROR`：异常中断 

---

### 6.2 报价单流程状态
建议定义：

+ `NEW`：新抓取 
+ `WAIT_MANUAL_PRICE`：待人工补价 
+ `AUTO_PRICING`：自动补价中 
+ `WAIT_SUBMIT`：待提交 
+ `SUBMITTING`：提交中 
+ `COMPLETED`：已完成 
+ `PARTIAL_EXCEPTION`：部分异常 
+ `FAILED`：处理失败

## 11. 异常处理建议
### 11.1 接单服务异常
+  超过心跳阈值未更新状态时，标记服务异常 
+  首页异常提醒卡片展示“接单服务异常中断” 

### 11.2 详情抓取失败
+  写入 `online_order_kaisi_execution_log`
+  若达到重试次数，报价单状态改为 `FAILED`

### 11.3 自动补价失败
+  明细状态改为 `WAIT_MANUAL_PRICE`
+  首页异常提醒可展示“补价完成仍存在未匹配价格” 

### 11.4 提交失败
+  写入 `online_order_kaisi_submit_item`
+  报价单状态保持 `WAIT_SUBMIT` 或 `PARTIAL_EXCEPTION`
+  支持前端再次手动触发提交

## 13. 总结
本设计方案在现有 SQL 表基础上，完成了以下增强：

1.  将 Flask 服务明确定位为 **接单控制与队列执行层**
2.  使用 Redis 实现 **队列解耦**
3.  使用后端服务承接 **状态同步、日志持久化、前端展示**
4.  明确了首页、报价单管理、人工补价、历史报价单四个页面的结构 
5.  明确了接单流程、人工补价流程、提交流程 
6.  补充了接单任务表、任务日志表、价格快照表、用户开思配置表等关键设计
